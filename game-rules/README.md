# Game Rules Directory

这个目录用于存放 `offline-cards` 各个卡牌游戏的**规则基线**、**当前实现边界**、以及**给 Jules / reviewer / 后续开发参考的判定标准**。

它的目标不是写成百科全书，而是让每个游戏都拥有一份：

- 可持续维护的规则说明
- 可执行的实现边界
- 可引用的交互约束
- 可复用的验收样例

---

## 这个目录是给谁看的
主要面向：

1. **Jules**
- 在创建或跟进某个游戏任务时，作为该游戏的固定规则参考
2. **offline-cards-reviewer**
- 在巡检、提需求、验收 PR 时，作为判断依据
3. **项目维护者**
- 在继续扩展规则、修 bug、做重构时，作为长期约束文档
4. **未来开发者**
- 快速理解某个游戏当前到底“实现到什么程度”

---

## 这个目录不是什么
它**不是**：

- 临时聊天记录
- 每日巡检日志
- 面向玩家的完整规则百科
- 所有历史讨论的堆积区

这些内容应分别放在：

- `memory/`：每日记忆、聊天修正、短期事实
- `projects/offline-cards-review.md`：连续巡检结论与优先级
- `projects/offline-cards-jules-guide.md`：发给 Jules 的长期产品/设计/模块化约束

## 命名建议

文件名尽量：

- 小写
- 英文
- 用连字符连接
- 一个文件对应一个明确规则范围

推荐示例：

- `uno.md`
- `zsy.md`
- `sanguosha-standard.md`
- `sanguosha-1v1.md`
- `doudizhu.md`

不推荐：

- `newrules.md`
- `sgs-final2.md`
- `玩法说明最新版.md`

因为这些名字不利于长期维护和程序化引用。

## 每个规则文件建议包含什么

每个游戏规则文件，建议至少包含以下章节。

### 1. Scope / 范围
说明这份文件覆盖什么，不覆盖什么。

例如：
- 标准版 / 基础版
- 是否含扩展包
- 是否只做最小可玩
- 是否暂不实现复杂边界规则

---

### 2. Goal / 当前目标
说明当前项目阶段想实现到什么程度。

例如：
- 最小可玩
- 核心回合闭环可跑通
- 支持基础多人对局
- 暂不追求完整赛事级规则还原

这一节非常重要，可以防止 Jules 一次性过度扩展。

---

### 3. Turn Structure / 回合结构
把回合拆清楚。说明回合如何推进，以及每个阶段允许什么动作。


### 4. Core Rules / 核心规则
写清楚该游戏最关键的基础规则，不必一开始就追求穷尽所有边角情况。

---

### 5. Interaction Mapping / 交互映射
说明这些规则在产品中应该如何表现。

例如：
- 哪些动作需要确认
- 哪些动作需要选目标
- 哪些阶段需要锁按钮
- 哪些状态要明确提示给玩家

---

### 6. Current Implementation Boundary / 当前实现边界
明确当前已实现什么、暂不实现什么，避免任务发散。

---

### 7. Acceptance Examples / 验收样例
给出若干端到端样例，帮助开发和验收判断“是否真的可用”。

---

## 推荐写法原则

### 1. 先写够用版本
先写当前项目真正需要的规则和边界，不必一开始就写成完整百科。

### 2. 区分规则与实现
建议区分：
- 原始/标准规则是什么
- 当前项目阶段准备怎么实现

### 3. 多写可执行判断，少写空泛描述
例如优先写：
- “未满足条件前，确认按钮必须 disabled”
而不是：
- “这里的交互要更合理”

### 4. 保持模块化意识
如果某条规则会影响代码结构，也可以顺手写明：
- 哪些内容应属于游戏专属模块

这个目录的作用是：

> 把每个游戏需要长期引用的规则、实现边界和交互约束，沉淀成稳定的项目级文档。
